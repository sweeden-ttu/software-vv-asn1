Pipeline Architecture

Input (query + content_type)
    │
    ▼
┌──────────────────┐
│ classify_content  │  ← LLM auto-detects content type
└────────┬─────────┘
         │
    ┌────▼────┐
    │ ROUTER  │  ← Conditional edge based on content type
    └────┬────┘
         │
    ┌────┼────┬────┬────┬────┬────┐
    ▼    ▼    ▼    ▼    ▼    ▼    ▼
  NEWS JUDGE OFCL ELEC  LAW COURT TMPL   ← 7 validator nodes
    │    │    │    │    │    │    │
    └────┴────┴────┴────┴────┴────┘
         │
    ┌────▼─────────────┐
    │ aggregate_results │  ← Determine pass/fail/retry
    └────────┬─────────┘
         │
    ┌────▼────┐
    │ RETRY?  │  ← If failed & retries left → loop back
    └────┬────┘
         │
    ┌────▼────┐
    │   END   │  ← Output: verified/unverified + provenance
    └─────────┘

Each validator checks against authoritative APIs (CourtListener, Congress.gov, FEC) plus LLM cross-reference, producing weighted confidence scores with full provenance metadata.

Experiments

Experiment 1: Baseline Hallucination Rate

10 ground-truth legal citations tested against GPT-4o-mini without verification. Measures proportion of hallucinated/incorrect citations.

Experiment 2: Pipeline Effectiveness

Same citations run through validator pipeline. Measures precision (verified correct / all verified), recall (verified correct / all correct), and post-pipeline hallucination rate.

Experiment 3: LangGraph vs Post-Hoc

Compares (A) LangGraph with retry logic vs (B) simple LLM + post-hoc self-verification. Measures accuracy and latency trade-offs.

Experiment 4: Security Red-Team

10 adversarial tests: prompt injection, source spoofing (including homograph attacks), fabricated citations/officials/laws, data exfiltration attempts, and historical manipulation.

Conceptual Blog Posts

Three blog posts explore the theoretical and practical foundations underlying this project.

Blog 1: 2+2 No Longer Equals 4: The Wild New Frontier of AI Software Testing

This post establishes the fundamental paradigm shift from deterministic to non-deterministic software testing. It introduces the NIST framework's seven pillars of Trustworthy AI (Safety, Resilience, Explainability, Privacy, Fairness, Reliability, Accountability) which directly inform our validator pipeline design. Key concepts applied in this project include: the distinction between Validation (dynamic, running the pipeline) and Verification (static, formal proof of correctness); the rise of Agentic testing platforms like LangChain/LangSmith for tracing; and the critical role of Red Teaming for Prompt Injection — which is exactly what Experiment 4 implements.

Project connection: Our LangSmith tracing across all 7 validators implements the "peek inside the black box" approach described. Our EP tests represent the new functional testing paradigm for non-deterministic systems.

Blog 2: Architecting Safety: Why the Future of Enterprise AI is Graph-Based

This post argues that graph-based workflows (LangGraph) are the only viable path for enterprise AI safety. The "Blackboard" shared-state metaphor maps directly to our PipelineState TypedDict — a central state object that all validators read from and write to, with no direct environment mutation. The core insight — "the agent cannot go off the graph" — is precisely why our pipeline uses conditional routing through defined nodes rather than open-ended LLM chains. The Test Oracle Problem discussion explains why our validators use weighted confidence scores (domain trust + API verification + LLM cross-reference) rather than binary pass/fail.

Project connection: Our pipeline architecture (classify → route → validate → aggregate → retry/end) is the direct implementation of the graph-based safety architecture described. Each validator is a testable node with defined inputs and outputs.

Blog 3: The Mars Rover Metric: Engineering Reliability into Non-Deterministic AI Agents

This post connects classical SDLC principles to AI agent development. The Verification vs Validation distinction maps to our two testing approaches: structural tests (verification — checking the code paths) and EP functional tests (validation — checking the product meets requirements). The Mars Rover Metric — applying formal rigor to safety-critical components — explains why our court document and judge validators use authoritative API verification (CourtListener, PACER) rather than relying solely on LLM knowledge. The Requirement Traceability Matrix concept is implemented through our provenance metadata on every output.

Project connection: Temperature=0 for deterministic testing, JSON enforcement for parseable outputs, and our full provenance chain (source → verification_date → confidence → status) implement the Requirement Traceability described.

How to Run

# Set environment variables (or add to ~/.zprofile)
export OPENAI_API_KEY="sk-..."
export LANGSMITH_API_KEY="lsv2_pt_..."
export LANGCHAIN_TRACING_V2=true
export LANGCHAIN_PROJECT="Software Validation and Verification"

# Install dependencies
pip install langgraph langchain langchain-openai langsmith pytest pytest-cov requests

# Run tests
cd legal-luminary
python -m pytest tests/ -v                                    # All 62 tests
python -m pytest tests/ --cov=validators --cov=state          # With coverage

# Run pipeline demo
python pipeline.py                                            # 7 demo validations

# Run experiments
python experiments/exp1_baseline.py                           # Baseline hallucination
python experiments/exp2_pipeline_effectiveness.py             # Pipeline effectiveness
python experiments/exp3_validator_vs_posthoc.py               # Architecture comparison
python experiments/exp4_security_redteam.py                   # Red-team evaluation
